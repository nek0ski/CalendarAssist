========================================================================
MODULE SPECIFICATION: v0_polling (LONG-POLLING MONOLITHIC PROTOTYPE)
========================================================================

[1. ARCHITECTURE OVERVIEW]
This module houses the original stateful prototype of CalendarAssist. The 
runtime engine relies on a continuous background script loop (Long Polling) 
to actively query Telegram's master API servers for incoming update sequences. 

It is highly effective for localized debugging and initial implementation 
stages, but demands an uninterrupted 24/7 compute engine execution thread.

[2. PIPELINE GRAPH]
Unstructured Input -> Local Long-Polling Thread -> Gemini Core -> 
Pydantic Structural Constraints -> Ephemeral RAM Context -> Google API Mutation -> 
Native Async JobQueue (APScheduler Thread Blocks)

[3. SYSTEM MANIFEST & RESOURCE DECOUPLING]
- Entry Point: `bot.py` (Handles structural setup and loops).
- Routing Layer: `handlers.py` (Fires processing pipelines and callbacks).
- Runtime Dependencies: Requires the `[job-queue]` parameter enhancement 
  on the `python-telegram-bot` manifest package along with `apscheduler`.

[4. SPECIFIC ENVIRONMENT PARSING MATRIX]
System initialization within this directory checks your root `.env` layout 
or injection layers strictly for the following variables:

- `BOT_TOKEN`               -> Active Telegram bot client identifier string.
- `GEMINI_API_KEY`          -> Access token issued via Google AI Studio.
- `GOOGLE_CREDENTIALS_JSON` -> Serialized dictionary matching client credentials.
- `GOOGLE_TOKEN_JSON`       -> Autogenerated valid user authentication session block.

*Note: CRON_JOB_API_KEY and PYTHONANYWHERE_USERNAME parameters are safely 
ignored by this module's runtime process.*

[5. OPERATIONAL DEPENDENCIES & BEHAVIOR CONSTRAINTS]
- Memory Overhead: High reliance on volatile system state. User configuration 
  caching (`pending_event`) is managed directly inside python-telegram-bot's 
  internal user_data memory space. Processes that interrupt or reboot the script 
  will instantly wipe out active verification pipelines.
- Notification Lifecycle: Automation tasks (1-hour warnings) utilize internal 
  asynchronous scheduler execution queues (`context.job_queue.run_once`). If the 
  hosting container platform enters a sleep cycle, times out, or scales down, 
  all scheduled reminder tasks are dropped permanently.

[6. HOW TO DEPLOY AND RUN]
To spin up this stateful implementation, open your local workspace shell 
and execute:

    cd v0_polling
    python bot.py
